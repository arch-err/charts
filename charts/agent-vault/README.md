# agent-vault

Helm chart for [Infisical Agent Vault](https://github.com/Infisical/agent-vault) — an HTTPS forward proxy that injects upstream credentials on behalf of AI agents and interactive shells. Real secrets stay in the vault; actors only ever hold revocable handles.

> **Status:** upstream is research-preview as of 2026-04-22. Expect breaking changes between point releases.

## What this chart deploys

- `Deployment` running `infisical/agent-vault`
- `Service` exposing two ports:
  - `14321` — HTTP API + web UI
  - `14322` — TLS forward-proxy (terminated by agent-vault itself with per-host certs minted from the AV CA)
- `PersistentVolumeClaim` for `/data` (holds encrypted vault state and the CA private key)
- *(optional)* `TLSRoute` for the proxy port via Gateway API passthrough

The UI `HTTPRoute` is intentionally **not** generated by this chart — wire it through your platform/gateway convention so the UI hostname follows the same pattern as your other apps.

## Required values

| Key | Why |
|---|---|
| `masterPassword.existingSecret` | Name of a `Secret` in the same namespace with key `AGENT_VAULT_MASTER_PASSWORD`. Without this the pod will boot into interactive setup and never become Ready. |
| `externalAddr` | Public URL of the UI. Used in invite emails. |

## TLS proxy exposure

The proxy port (`14322`) **must not** be TLS-terminated upstream — clients need to see the per-host certificates that agent-vault mints from its own CA. Pick one of:

- **TLSRoute (Gateway API passthrough)** — set `tlsRoute.enabled: true`, point `parentRef` at a Gateway listener with `tls.mode: Passthrough`, and put the proxy hostname in `tlsRoute.hostname`. Clients use `HTTPS_PROXY=https://<handle>:vault@<hostname>:443`.
- **Service `type: LoadBalancer`** — set `service.type: LoadBalancer`. Cluster assigns an IP; clients use `HTTPS_PROXY=https://<handle>:vault@<lb-ip>:14322`.
- **Service `type: NodePort`** — same pattern but on a node IP.

## Master password — generation

Generate once, never touch again. Example:

```sh
LC_ALL=C tr -dc 'A-Za-z0-9' </dev/urandom | head -c 64
```

Store it in the password manager *and* commit a sops-encrypted (or sealed-secrets) Secret manifest that `masterPassword.existingSecret` references. Losing it makes the vault unrecoverable.

## CA cert distribution

After first boot, retrieve the AV CA cert from the pod and install it in a per-process trust scope on each client (do not add to the system trust store):

```sh
kubectl exec -n <ns> deploy/agent-vault -- cat /data/ca.crt > ~/.local/share/agent-vault-ca.crt
export SSL_CERT_FILE=~/.local/share/agent-vault-ca.crt
export HTTPS_PROXY=https://<handle>:vault@<proxy-host>:<port>
```
